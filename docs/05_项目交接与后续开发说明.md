## 3. 项目接收方运行步骤

以下命令按 Windows PowerShell 环境编写。

### 3.1 解压项目压缩包

将项目压缩包解压到本地目录，例如：

```powershell
D:\Projects\实训项目一
```

### 3.2 进入项目目录

```powershell
cd D:\Projects\实训项目一
```

### 3.3 创建后端虚拟环境

```powershell
cd backend
python -m venv .venv
```

### 3.4 安装后端依赖

```powershell
.venv\Scripts\activate
pip install -r requirements.txt
```

### 3.5 复制环境配置文件

回到项目根目录：

```powershell
cd ..
copy .env.example .env
```

### 3.6 修改数据库连接

使用编辑器打开 `.env`，将数据库连接改为本机 MySQL 密码：

```env
DATABASE_URL=mysql+pymysql://root:自己的MySQL密码@localhost:3306/gov_ai_assistant
```

### 3.7 创建 MySQL 数据库

在 MySQL 客户端或 DataGrip 中执行：

```sql
CREATE DATABASE gov_ai_assistant
  DEFAULT CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;
```

### 3.8 初始化数据库表

在项目根目录运行：

```powershell
python scripts/init_db.py
```

### 3.9 初始化政务分类

```powershell
python scripts/seed_categories.py
```

### 3.10 导入示例政策文档

```powershell
python scripts/import_knowledge.py
```

### 3.11 构建向量索引

```powershell
python scripts/build_vector_index.py
```

### 3.12 启动后端

```powershell
cd backend
.venv\Scripts\activate
uvicorn app.main:app --reload
```

### 3.13 启动前端

另开 PowerShell 终端，从项目根目录运行：

```powershell
cd frontend
npm install
npm run dev
```

### 3.14 运行一键检查脚本

后端启动后，另开 PowerShell 终端，回到项目根目录运行：

```powershell
python scripts/check_project.py
```

## 4. 环境配置说明


然后修改数据库连接：


2. 当前仓库没有提交数据库密码。
3. 当前仓库没有提交真实 API Key。
4. 当前仓库没有提交 `backend/.venv`。
5. 当前仓库没有提交 `frontend/node_modules`。
6. 当前仓库没有提交 Chroma 实际数据库文件。
7. `data/vector_db` 中只保留 `.gitkeep`。

## 5. 数据库初始化说明

首次运行项目前，需要先创建 MySQL 数据库：

```sql
CREATE DATABASE gov_ai_assistant DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

如果本机 MySQL 中已经存在 `gov_ai_assistant` 数据库，可以不用重复创建。

数据库表通过以下命令初始化：

```powershell
python scripts/init_db.py
```

该脚本使用 SQLAlchemy `create_all`，只会创建缺失表，不会删除已有表和已有数据。

## 6. 向量索引构建说明

在数据库表、政务分类和示例政策文档准备完成后，运行：

```powershell
python scripts/build_vector_index.py
```

该脚本会读取有效知识文档，执行文本切分，写入 `knowledge_chunks` 表，并将文本片段写入本地 Chroma collection。

如果一键检查脚本提示向量索引为空，也可以重新运行该命令。

## 10. 大模型 API 后续调试说明

当前项目只是预留 API 接入能力，默认不会真实调用大模型。

后续维护人员需要完成：

1. 确定使用 DeepSeek、通义千问或其他 OpenAI-compatible API。
2. 修改本地 `.env`。
3. 将 `LLM_ENABLE` 改为 `true`。
4. 填入 `LLM_PROVIDER`、`LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL`。
5. 完善 `backend/app/services/llm_service.py` 中的真实调用逻辑。
6. 调试 `/llm/status`。
7. 调试 `/chat`，使 `answer_mode` 从 `template_rag` 变为 `llm_rag`。
8. 确认敏感信息不会完整发送给大模型。

建议调试顺序：

1. 先确认 `.env` 配置能被后端读取。
2. 再访问 `GET /llm/status`，确认 `enabled=true` 且 `available=true`。
3. 最后测试 `POST /chat`，确认回答可以基于知识库检索结果调用大模型生成。

## 11. Git 协作注意事项

1. 不要提交 `.env`。
2. 不要提交 `backend/.venv`。
3. 不要提交 `frontend/node_modules`。
4. 不要提交 `data/vector_db/chroma.sqlite3`。
5. 不要提交 `data/vector_db` 下的 UUID 文件夹。
6. 不要直接 `git add .`。
7. 每次提交前先 `git status`。
8. 每完成一个功能先验证再 `commit`。

建议每次提交只添加本次相关文件，例如：

```powershell
git status
git add docs/05_项目交接与后续开发说明.md
git commit -m "更新项目交接与后续开发说明"
```
